Users¶

API Keys¶

pulsar supports token (API key) authentication. API keys are generated on request, and must be manually granted permissions (blank for full permissions). To authenticate with an API key, send it in the request header as Authorization: Token a-long-api-key.

POST /api_keys¶

Creates an API key for use. Keys are unrecoverable after generation; if a key is lost, a new one will need to be generated.

Example request:

POST /api_keys HTTP/1.1

{
  "username": "lights",
  "password": "12345",
  "permanent": false,
  "timeout": 7200
}

Example response:

{
  "status": "success",
  "response": {
    "hash": "abcdefghij",
    "key": "abcdefghijklmnopqrstuvwx",
    "permissions": [
      "api_keys_view",
      "invites_send"
    ]
  }
}
Response JSON Array of Objects:
 
  • hash (string) – The identification id of the API key
  • key (string) – The full API key
  • permissions (list) – A list of permissions allowed to the API key, encoded as str
Status Codes:
  • 200 OK – Successfully created API key
DELETE /api_keys¶

Revokes all API keys currently in use by the user. Requires the api_keys_revoke permission to revoke one’s own API keys, and the api_keys_revoke_others permission to revoke the keys of other users.

Example response:

{
  "status": "success",
  "response": "All api keys have been revoked."
}
Status Codes:
  • 200 OK – Successfully revoked API keys
  • 403 Forbidden – User does not have permission to revoke API keys
GET /api_keys¶

View all API keys of a user. Requires the api_keys_view permission to view one’s own API keys, and the api_keys_view_others permission to view the API keys of another user.

Example response:

{
  "status": "success",
  "response": [
     "<APIKey>",
     "<APIKey>"
  ]
}
Query Parameters:
 
  • include_dead (boolean) – Include dead (previously used) API keys
Response JSON Object:
 
  • response (list) – A list of API keys
Status Codes:
  • 200 OK – Successfully viewed API keys
  • 403 Forbidden – User does not have permission to view user’s API keys
  • 404 Not Found – User does not exist
DELETE /api_keys/(hash)¶

Revokes an API key currently in use by the user. Requires the api_keys_revoke permission to revoke one’s own API keys, and the api_keys_revoke_others permission to revoke the keys of other users.

Example request:

DELETE /api_keys HTTP/1.1

{
  "hash": "abcdefghij"
}

Example response:

{
  "status": "success",
  "response": "API Key abcdefghij has been revoked."
}
Request JSON Object:
 
  • hash (str) – The hash of the API key
Status Codes:
  • 200 OK – Successfully revoked API keys
  • 404 Not Found – API key does not exist or user does not have permission to revoke the API key
GET /api_keys/(hash)¶

View info of an API key. Requires the api_keys_view permission to view one’s own API keys, and the api_keys_view_others permission to view the API keys of another user.

Example response:

{
  "status": "success",
  "response": "<APIKey>"
}
Response JSON Object:
 
  • response (dict) – An API key
Status Codes:
  • 200 OK – Successfully viewed API key.
  • 404 Not Found – API key does not exist.

Invites¶

POST /invites¶

Sends an invite to the provided email address. Requires the invites_send permission. If the site is open registration, this endpoint will raise a 400 Exception.

Example request:

POST /invites/an-invite-code HTTP/1.1

{
  "email": "bright@puls.ar"
}

Example response:

{
  "status": "success",
  "response": "<Invite>"
}
Request JSON Object:
 
  • email (string) – E-mail to send the invite to
Status Codes:
  • 200 OK – Successfully sent invite
  • 400 Bad Request – Unable to send invites or incorrect email
  • 403 Forbidden – Unauthorized to send invites
GET /invites¶

View sent invites. If a user_id is specified, only invites sent by that user will be returned, otherwise only your invites are returned. If requester has the invites_view_others permission, they can view sent invites of another user.

Example response:

{
  "status": "success",
  "response": [
     "<Invite>",
     "<Invite>"
  ]
}
Query Parameters:
 
  • used (boolean) – (Optional) Whether or not to only show used invites (overrides include_dead)
  • include_dead (boolean) – (Optional) Whether or not to include expired invites
Response JSON Object:
 
  • response (list) – A list of invites
Status Codes:
  • 200 OK – View successful
  • 403 Forbidden – User does not have permission to view user’s invites
DELETE /invites/(code)¶

Revokes an active invite code, preventing it from being used. The invite is returned to the user’s account. Requires the invites_revoke permission to revoke one’s own sent invite, and the invites_revoke_others permission to revoke another user’s invites.

Example request:

DELETE /invite HTTP/1.1

{
  "id": "an-invite-code"
}

Example response:

{
  "status": "success",
  "response": "<Invite>"
}
Status Codes:
  • 200 OK – Revocation successful
  • 403 Forbidden – Unauthorized to revoke invites
  • 404 Not Found – Invite does not exist or user cannot view invite
GET /invites/(code)¶

View the details of an invite. Requires the invites_view permission. Requires the invites_view_others permission to view another user’s invites.

Example request:

GET /invite HTTP/1.1

{
  "code": "an-invite-code"
}

Example response:

HTTP/1.1 200 OK
Vary: Accept
Content-Type: application/json

{
  "status": "success",
  "response": "<Invite>"
}
Status Codes:
  • 200 OK – View successful
  • 404 Not Found – Invite does not exist or user cannot view invite

Settings¶

PUT /users/settings¶

Change a user’s settings. Requires the users_edit_settings permission. Requires the users_moderate permission to change another user’s settings, which can be done by specifying a user_id.

Example request:

PUT /users/settings HTTP/1.1

{
  "existing_password": "y-&~_Wbt7wjkUJdY<j-K",
  "new_password": "an-ev3n-be77er-pa$$w0rd"
}

Example response:

{
  "status": "success",
  "response": "Settings updated."
}
JSON Parameters:
 
  • existing_password (string) – User’s existing password, not needed if setting another user’s password with moderate_user permission.
  • new_password (string) – User’s new password. Must be 12+ characters and contain at least one letter, one number, and one special character.
Status Codes:
  • 200 OK – Settings successfully updated
  • 400 Bad Request – Settings unsuccessfully updated
  • 403 Forbidden – User does not have permission to change user’s settings

Users¶

POST /users¶

Creates a user account with the provided credentials. An invite code may be required for registration.

Example request:

POST /register HTTP/1.1

{
  "username": "user_one",
  "password": "y-&~_Wbt7wjkUJdY<j-K",
  "email": "user_one@pul.sar",
  "code": "my-invite-code"
}

Example response:

{
  "status": "success",
  "response": {
    "username": "lights"
  }
}
JSON Parameters:
 
  • username – Desired username: must start with an alphanumeric character and can only contain alphanumeric characters, underscores, hyphens, and periods.
  • password – Desired password: must be 12+ characters and contain at least one letter, one number, and one special character.
  • email – A valid email address to receive the confirmation email, as well as account or security related emails in the future.
  • code – (Optional) An invite code from another member. Required for registration if the site is invite only, otherwise ignored.
Response JSON Array of Objects:
 
  • username (string) – username the user signed up with
Status Codes:
  • 200 OK – registration successful
  • 400 Bad Request – registration unsuccessful
GET /users/(int: user_id)¶

Return general information about a user with the given user ID. If the user is getting information about themselves, the API will return more detailed data about the user. If the requester has the users_moderate permission, the API will return even more data.

Example response:

HTTP/1.1 200 OK
Vary: Accept
Content-Type: application/json

{
  "status": "success",
  "response": "<User>"
}
Status Codes:
  • 200 OK – User exists
  • 404 Not Found – User does not exist

Moderation¶

PUT /users/(int: user_id)¶

Moderate a user - change password for them, alter stats, modify basic permissions, etc.

Example request:

PUT /users/1/moderate HTTP/1.1

{
  "password": "an-ev3n-be77er-pa$$w0rd"
}

Example response:

HTTP/1.1 200 OK
Vary: Accept
Content-Type: application/json

{
  "status": "success",
  "response": "<User>"
}
JSON Parameters:
 
  • password (string) – User’s new password. Must be 12+ characters and contain at least one letter, one number, and one special character.
Status Codes:
  • 200 OK – User successfully moderated
  • 400 Bad Request – User unsuccessfully moderated
  • 403 Forbidden – User does not have permission to moderate some parts of user
